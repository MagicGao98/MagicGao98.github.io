软件工程(十)_软件测试

软件测试基础

在测试阶段测试人员努力设计出一系列测试案例，目的是”破坏”已建好的系统，竭力证明程序中有错误，不能按照预定要求正常工作

测试阶段的根本目的是尽可能多地发现并排除软件中潜在的错误，最终把高质量的软件系统交给用户使用

测试准则：

  1. 所有测试都应该能追溯到用户需求
  2. 应该远在测试开始之前就制定出测试计划
  3. 把Pareto原理应用到软件测试中，Pareto原理说明，测试发现的错误中80%很可能是由程序中的20%的模块引起的
  4. 应该从小规模测试开始，再逐步进行大规模测试
  5. 穷举测试是不可能的，测试只能证明程序中存在错误，不能证明程序中没有错误
  6. 为了达到最佳的测试效果，应该有独立的第三方从事测试工作

测试方法

黑盒测试：把程序看作一个黑盒子，步烤炉程序的内部结构和处理过程；黑盒测试是在程序接口进行测试，只检测是否符合规格说明书；黑盒测试又称功能测试

白盒测试：把程序看作装在一个透明的白盒子里，测试程序内部的主要执行通路是否按要求正常工作；白盒测试也称结构测试

测试步骤

  1. 模块测试

    模块测试的目的是保证每个模块作为一个单元能正常工作，模块测试也称单元测试，这个测试往往发现的是编码和详细设计阶段的错误

  2. 子系统测试

    把单元测试通过的模块组装成子系统，检测模块间是否能够正常通信和协调，这个步骤主要是测试模块的接口

  3. 系统测试

    把测试通过的子系统装配成一个完整的系统来测试，确定系统能够实现需求说明书上指定的功能，并且动态特征符合预定要求；子系统测试和系统测试通常都成为集成测试

  4. 验收测试

    在用户的参与下，测试整个系统，确定系统功能符合用户的需求；验收测试也称确认测试

  5. 平行运行

    新旧系统平行运行一段时间，防止新系统有未检测到的错误

单元测试

测试重点：模块接口、局部数据结构、重要的执行通路、出错处理通路、边界条件

代码审查：由审查小组(组长、设计者、编写者、测试者)进行代码审查

用计算机的测试方法发现错误之后，通常需要先改正错误之后才能进行后续的继续测试工作；模块内聚程度高可以简化单元测试过程

集成测试

集成测试的方式有两种：非渐增式测试和渐增式测试

非渐增式测试：先分别测试每个模块，再将所有模块集成起来再测试整个系统，即一次性集成

渐增式测试：每测试完一个模块之后就将该模块集成到系统中，并对系统进行测试，逐步集成

确认测试

确认测试也称验收测试，验证软件的有效性

确认测试必须用户积极参与，确认测试通常进行黑盒测试

确认测试的一个重要内容是复查软件配置，保证软件配置的所有成分都齐全，质量符合要求，文档与程序完全一致具有完成软件维护所必须的细节，并且已经编好目录

Alpha测试：由用户在开发者的场所进行

Beta测试：由软件的最终用户在一个或多个客户场所进行，通常开发人员不在测试现场

白盒测试技术

设计测试用例(测试数据+预期的输出结果)是测试阶段的关键技术问题

白盒测试的方法很多，主要介绍逻辑覆盖和控制结构测试

逻辑覆盖：

  • 语句覆盖

    每条语句都至少执行一遍，语句覆盖是很弱的逻辑覆盖标准

  • 判断覆盖

    每条语句都至少执行一遍，并且每个判断的每一种可能都至少执行一次，注重整个判断的真或假；比语句覆盖强，但仍很弱

  • 条件覆盖

    语句覆盖基础上，判断表达式中的每一个条件的每种情况都要取一遍，注重每个条件的真或假；条件覆盖不包含于判断覆盖，判断覆盖也不一定包含于条件判断

  • 判断/条件覆盖

    判断表达式中的每个条件都取到各种情况的值，每个判断表达式也都取到各种可能结果

  • 条件组合覆盖

    要求选取足够多的测试数据，使得每个判断表达式中的判断条件的各种组合都至少出现一次，注重判断式内部的条件的组合，不是全部判断式条件的组合；条件组合覆盖一定满足判断覆盖、条件覆盖和判断/条件覆盖；条件组合覆盖并不是使程序中的每一条途径都执行到

  • 点覆盖

    选取足够多的测试数据，使得程序的执行路径至少经过流图的每一结点一次；标准和语句覆盖相同

  • 边覆盖

    使得程序执行路径至少经过流图的每个边一次；通常和判断覆盖一致(判断式只有一个判断条件时一致，多个时不一致)

  • 路径覆盖

    使得程序的每条可能的执行路径至少执行一次

控制结构测试：

  • 基本路径测试

    步骤：

    1. 根据程序设计结构画出相应的流图

    2. 计算流图的环形复杂度

    3. 确定线性独立路径的基本集合

      独立路径是指至少引入程序的一个新的处理语句集合或是一个新的条件的路径，流图中即是至少有条新边

    4. 设计强制执行基本集合中每条路径的测试用例，且每个条件在执行的时候都分别取真、假

  • 条件测试

  • 循环测试

    专注于测试循环结构的有效性，循环结构分三种：简单循环、串接循环和嵌套循环

黑盒测试技术

常见错误：

  1. 功能不正确或遗漏
  2. 界面错误
  3. 数据结构错误或外部数据库访问错误
  4. 性能错误
  5. 初始化或终止错误

黑盒测试技术很多，主要介绍等价划分、边界值分析和错误推断

等价划分：

有效等价类：对于程序规格说明合理的有意义的输入数据构成的集合

无效等价类：与有效等价类定义相反定义

设计测试用例之前，先根据程序的功能说明，将输入数据划分为有效等价类和无效等价类

等价类划分的启发式规则：

  1. 规定输入值范围时，划分一个有效等价类(范围内)，两个无效等价类(范围外的两端)
  2. 规定输入数据个数时，划分一个有效等价类(符合个数)，两个无效等价类(多余要求个数和少于要求个数)
  3. 规定输入一组值，且程序对输入值做不同处理时，每个处理允许输入数据是一个有效等价类，任一无效输入值都属于一个无效等价类，即多个有效等价类，一个无效等价类
  4. 规定输入数据不许遵守一定规则时，遵守规则的输入数据是一个有效等价类，和若干无效等价类(从不同违反角度划分)
  5. 规定输入数据为整数时，划分一个无效等价类，和正整数、零、负整数三个有效等价类

启发式规则很多，主要是凭着感觉走

每设计一个测试用例都必须包含一个新的等价类，直到所有等价类都覆盖到后，设计的测试用例才算完整

设计有效测试用例时，使其尽可能多的覆盖尚未覆盖的有效等价类，直到有效等价类覆盖完，一般只用一个测试用例

设计无效测试用例时，每次只覆盖一个无效等价类，直到无效等价类覆盖完，一般有多少个无效等价类就至少有多少个无效测试用例

边界值分析：

定义：

边界值分析法就是对输入或输出的边界值进行测试的一种黑盒测试方法。通常边界值分析法是作为对等价类划分法的补充，这种情况下，其测试用例来自等价类的边界

常见的边界值：

16bit：-32768~32767；

32bit：-65536~65535;

数组元素的第一个和最后一个；

循环的第0次、第1次和倒数第2次、最后一次

错误推断：

基本思想：列举出程序中所有可能的错误和容易发生错误的特殊情况，更具它们选择测试用例

调试

调试是在测试结束发现错误之后排除错误的过程

工作：确定程序中的可以错误的确切性质和位置，对程序(设计、编码)进行修改，排除这个错误

调试途径：蛮干法、回溯法、原因排除法(对分查找法、归纳法和演绎法)

调试过程：

  1. 从错误的外部表现形式入手，确定程序中的出错位置
  2. 研究相关程序，找出错误的内在原因
  3. 修改设计和代码，排除这个错误
  4. 重复进行暴露这个错误的原始促使和某些相关测试

注意事项：

  1. 有头脑去分析思考与错误相关的信息而不是用脚趾头
  2. 调试工具只是工具，可以用但不能依赖
  3. 试探法是下下策
  4. 不要改一个Bug出现两个Bugs

软件可靠性

软件可靠性：在给定的时间间隔内，按照规格说明书的规定成功地运行的概率

软件可使用性：程序在给定的时间点，按照规格说明书的规定，成功运行的概率

IEEE给出的错误的含义：由开发人员造成的软件差错(bug)

IEEE给出的故障的含义：由错误引发的软件的不正确的行为
