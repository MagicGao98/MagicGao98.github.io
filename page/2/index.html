面向数据流的分析方法

目的： 给出系统化的设计软件结构的途径，数据流图—>结构图

软件需求分析阶段，信息流是关键，常用数据流图来描绘信息在系统中的加工和流动情况，面向数据流的分析方法就是建立从数据流到软件结构的映射

任何系统都可以用数据流图来表示，面向数据流的设计方法理论上可设计任何软件的结构，信息流的类型决定了映射的方法，信息流类型包括：变换流和事物流

变换流：信息沿着输入通路进入系统，同时从外部形式转换为内部形式，进入系统的信息通过变换中心，经过加工处理后再沿着输出通路变换成外部信息离开软件系统，这种信息流就叫作变换流

事务流：基本系统模型的信息流原则都可以划分为变换流，但是如果信息流是以事务为中心的话，即接收一个输入数据（称为：事务）之后分析每个事务类型并选择一条活动通路进行数据处理。如此，就将这样特殊数据流称作事务流

面向数据流的分析过程：

  • 变换分析

    1. 复查基本系统模型

    2. 复查并精化数据流图

      保证数据流图中每个处理都代表一个规模适中相对独立的子功能

    3. 确定数据流图具有变换特征还是事务特征

    4. 确定输入流和输出流边界，从而孤立出变换中心

    5. 完成一级分解

      产生最顶层的控制模块 Cm ，以及下列从属模块：

      输入信息处理控制模块Ca ，协调对所有输入数据的接收

      变换中心控制模块Ct ，管理内部形式的数据的所有操作

      输出信息处理控制模块Ce ，协调输出信息的产生过程

    6. 完成二级分解

      从变换中心开始逆着输入通路向外移动，把输入通路中的每个处理映射成软件结构中Ca 控制下的一个低层模块；然后沿着输入通路向外移动，把输出通路中每个处理映射成直接或间接接收Ce 控制的一个低层模块；最后把变换中心内的每个处理映射成Ct 下的一个模块

    7. 使用度量或启发式规则对第一次分割得到的软件结构进一步精化

      根据模块独立化原理，对初步划分的模块进行再分解或合并

  • 事务分析

    步骤和变换分析步骤大部分相同或类似，主要区别是映射方法不同

    由事务流映射成的软件结构包括一个接收分支和一个发送分支，接收分支和变换分析产生的输入结构的映射方法类似，即从事务中心的边界开始，沿着逆向接收信息通路的处理映射成模块；发送分支的结构包含一个调度模块，和它控制的下层所有活动模块

对象模型

对象模型标识静态的、结构化的系统的数据性质，是对模拟客观世界实体的对象以及对象彼此之间的关系的映射，描述系统的静态结构。通常使用UML提供的类图进行对象模型建模

步骤：

  1. 确定对象类和关联，对于大型复杂问题可以划分若干主题

    确定对象类：采用非正式分析：以需求陈述为依据，将名词作为类和对象的候选者，把形容词作为确定属性的线索，把动词作为服务(操作)的候选者；再在非正式分析的基础上，去掉冗余、无关、笼统的名词，辨别属性、操作相关的名词、暂时忽略和实现相关的类和对象

    确定关联：

  2. 给类和关联添加属性

  1. 利用适当的继承关系进一步合并和组织类

  • 类图

    描述类于类之间的静态关系，它时创建其他UML图的基础，一个系统可以由几张类图构成，一各类可以出现在几张类图中；类图由类和类于类之间的关系组成，类于类之间的关系包括：关联、聚集、泛化（继承）、依赖和细化

动态模型

步骤：

  1. 编写经典交互行为的脚本，至少保证不遗漏常见交互行为
  2. 从脚本中提取事件，确定触发每个事件的动作对象以及接受事件的目标对象
  3. 排列事件发生的的次序，确定每个对象可能有的状态以及状态之间的转换关系，并用状态图描述出来
  4. 比较各个对象的状态图，检查它们之间的一致性，确保事件之间的匹配

脚本：指系统在某一执行期间内出现的一系列事件

脚本描述用户与目标系统之间的一个或多个典型的交互过程，以便对目标系统的行为进行更具体的认识；脚本描述事件序列，每当系统中的对象与用户交换信息时，就发生一个事件，对于每个事件都应该指明事件发生的触发对象和目标对象以及该事件的参数

  • UML顺序图

    顺序图是一种详细表示对象之间以及对象与参与者之间交互的图，它由一组协作的对象（或参与者）以及他们之间可发送的消息组成，强调消息之间的顺序

    顺序图是二维的，垂直方向表示时间，水平方向表示不同的对象或参与者

    顺序图中的主要元素包括：

    • 对象或参与者
    • 生命线
    • 控制焦点
    • 消息

    参与要描述的消息交互场景中的对象和参与者依次排列在水平方向上，顺序不重要，消息发送者一般在最左侧，对象或参与者在顺序图中不一定在同一排上，要求所有在通信开始之前就存在的对象放置在图的顶部，在中途被创建的对象，放置在箭头指向的位置

    生命线用虚线表示，位于对象符号之下，表示对象在一段时间内的存在，在所有通信结束完成后人存在的对象生命线要延伸超过图中最后一个箭头线；如果对象在中途销毁，要在生命线的底端打×，这是针对自析构的情况

    控制焦点表示对象的一个操作执行的持续时间，也称作激活；在顺序执行的情况下，控制焦点代表对象的操作或者他的从属操作活动着的持续时间；用一个窄长的矩形表示控制焦点；对象操作递归调用自己或调用对象本身对象的其他操作时，控制焦点表示如下图：

    消息是对象之间通信的描述，一个消息的执行会引起一个操作的调用、发送一个信号、或引起一个对象被创建或删除；消息的接收对象可能会像调用者返回一个结果；消息用一条生命线到另一条生命线的一条带箭头的水平实线表示(简称箭线)，从源对象指向目标对象；需要时间发送的消息用倾斜的实线

功能模型

功能模型表示变化的系统的功能性质，指明系统应该做什么，功能模型由一组数据流图组成，UML中的用例图也是进行需求分析和建立功能模型的强有力工具，用例图建立起来的功能模型也称用例模型，一个用例模型由若干幅用例图组成

  • 用例图

    步骤：

    1. 定义系统
    2. 寻找行为者和用例
    3. 描述用例
    4. 定义用例之间的关系
    5. 确定模型

STAR法则

STAR法则简介

STAR法则是情境(situation)、目标（target）、行动(action)、结果(result)四项的缩写。STAR法则是一种常常被面试官使用的工具，用来收集面试者与工作相关的具体信息和能力。STAR法则比起传统的面试手法来说，可以更精确地预测面试者未来的工作表现

STAR法则也可以用来工作总结、描述事情或者项目经历，使得叙述更完善和有逻辑性

STAR法则实际应用

Situation背景：为什么要做这样的一个事情/工作/项目？

Task任务：主要的职责和目标是什么？

Action行动：为了完成工作，做了些什么努力？

Result成果：通过行动，工作上取得了哪些成果？列出具体数据

注意事项

使用STAR法则时，每点要提炼出关键词，最好使用一句话叙述，避免一段一段地叙述

  git config --global user.email '1039932706@qq.com

  • 添加文件夹中的文进入Git仓库

      git add <file

  • 查看仓库状态

  # git 不会识别空文件夹

    2、添加远程仓库的对接

      git remote add origin <远程仓库地址

    3、检测是否与远程仓库连接上

      git remote
  # 结果为origin 连接上了

    4、将本地的仓库推送 到远程仓库中

      git push -u origin master

F&Q

  • 如果出现Permission denied(publickey)，解决办法

    1、删除原来的origin

      git remote rm origin

    2、重新建立连接

      git remote add origin  https://github.com/xxx/xxxxx.git
  # 第一个XXX代表你的用户名，第二个XXX代表你push到github上的版本库名称
  # 执行这句命名时，会弹出登陆GitHub的窗口，登陆即可

    3、重新推送到远程仓库

      git push -u origin master

git clone项目时遇见git@github.com Permission denied (publickey)

问题重述

git@github.com: Permission denied (publickey). Could not read from remote repository.

产生原因

  • 客户端与服务器端未生成ssh key
  • 客户端与服务器端的ssh key不匹配

解决方案

重新生成一次ssh key，服务器端也重新配置一次

客户端生成 ssh key

# 客户端生成 ssh key
ssh-keygen -t rsa -C "yourmail@example.com"
# 直接一路回车
ssh -v git@github.com
# 最后两句出现
# No more authentication methods to try.  
# Permission denied (publickey).
ssh-agent -s
# 出现 SSH_AUTH_SOCK SSH_AGENT_PID等log信息
ssh-add ~/.ssh/id_rsa
# 直接回车，Identity added:...这是 ssh key 文件路径信息

配置服务端

打开~/.ssh/下的id_rsa.pub文件，复制其内容，进入你的GitHub账号，在setting 下，SSH and GPG key 下 new SSH key，然后id_rsa.pub文件内容复制到 key 中，完成后Add SSH key

验证 Key

ssh -T git@github.com
# 提示 Hi xxx! You've successfully authenticated....则成功解决问题

Q&A

执行ssh-add时出现 Could not open a connection to authentication agent

解决方法：执行ssh-add之前先执行：ssh-agent bash

虚拟机 Tips

  1. 虚拟机开机出现黑屏解决办法

    1. 以管理员身份运行cmd
    2. 在命令提示符窗口中输入“netsh winsock reset”，然后重启计算机即可解决。
    3. 如果不行，在VM->Settings->Hardware->Display在右面的内容栏中将Accelerate 3D graphics 取消打勾，然后重启电脑即可。

  2. VM 虚拟机共享文件夹

    1. 在VMware 的设置里面把虚拟机的文件共享开启，并设置需要开启的文件夹
    2. 在对应的虚拟机里面挂载好共享文件夹
      sudo vmhgfs-fuse .host:/  /mnt/hgfs
      完成就可用root用户在/mnt/hgfs/ 下找到共享的文件夹了

  3. vm打开虚拟机黑屏处理方法

    1. 管理员身份运行cmd
    2. 输入netsh winsock reset
    3. 重启计算机
      如果还不行，就在VM->setting->hardware->display的右面的内容栏中将Accelerate 3D graphics取消，然后重启计算机
